# Speed Tier Spreadsheet UI Spec

## Goal

Build a fast, readable speed-tier spreadsheet for Pokemon Champions. The UI should answer one question quickly:

```text
What Pokemon can reach this speed, and what conditions make that speed true?
```

The table is a speed index, not an editable spreadsheet. Prioritize scan speed, deterministic grouping, and clear modifier provenance.

## Data Source

Use grouped speed tier JSON generated by `@outspeedme/ingestor`.

Primary shape:

```ts
type SpeedTier = {
  speed: number;
  pokemon: SpeedTierPokemon[];
};

type SpeedTierPokemon = {
  id: number;
  pokedexNo: number;
  name: string;
  spread: SpeedSpread;
  effects: SpeedEffect[];
  finalSpeed: number;
};

type SpeedSpread = {
  nature: "neutral" | "positive" | "negative";
  evs: 0 | 252;
  ivs: number;
  level: number;
  rawSpeed: number;
};

type SpeedEffect = {
  kind: "ability" | "item" | "field" | "weather" | "terrain" | "stage";
  source: string;
  label: string;
  multiplier?: number;
  stage?: number;
  condition?: string;
};
```

Do not infer speed effect meaning in the frontend. Render `effects[]` as emitted by the ingestor.

## Table Columns

Use exactly four core columns:

```text
Speed | Pokemon | Spread | Effects
```

### Speed

Final actual speed. This is the grouping column.

Rules:

- Render once per speed group.
- Make it the strongest visual anchor.
- Use tabular numbers.
- Keep sticky on horizontal scroll if the table becomes wide.
- Sort descending by default.

Example:

```text
308
```

### Pokemon

Pokemon display name.

Rules:

- One row per Pokemon setup inside each speed group.
- Same-speed Pokemon share one speed group.
- Pokemon name should be readable before chips or secondary details.
- Use `pokedexNo` only for optional debugging/tooltips, not default display.

Example:

```text
Excadrill
```

### Spread

Compact nature + EV summary.

Display mapping:

```ts
positive -> "+Spe"
neutral -> "neutral"
negative -> "-Spe"
```

Format:

```text
+Spe 252
neutral 252
-Spe 0
```

Default table should not show IV, level, or raw speed. Put those in expanded details.

### Effects

Typed chips explaining all speed-changing context.

Rules:

- If no effects, render muted `baseline` or an empty dash.
- Always show multiplier chips before source chips.
- Effects must preserve source and condition clarity.
- Stacked effects should be readable without needing a tooltip.

Examples:

```text
[2x] [Sand Rush] [sand]
[1.5x] [Choice Scarf]
[2x] [Swift Swim] [rain] [1.5x] [Choice Scarf]
[1.5x] [Speed Boost] [after 1 turn]
```

Chip rendering:

```ts
if effect.multiplier exists -> multiplier chip
effect.label -> source chip
if effect.condition exists -> condition chip
if effect.stage exists -> stage chip, optional if label already communicates it
```

Do not render one vague `Modifier` cell. The whole point is provenance.

## Group Layout

Visual row model:

```text
Speed  Pokemon            Spread       Effects
308    Excadrill          +Spe 252     [2x] [Sand Rush] [sand]

290    Venusaur           +Spe 252     [2x] [Chlorophyll] [sun]
       Lycanroc           +Spe 0       [2x] [Sand Rush] [sand]

222    Mega Aerodactyl    +Spe 252     baseline
```

Implementation option:

- Use shadcn/ui `Table` components as the primary structure.
- Render one `TableRow` per Pokemon setup.
- Render speed once per group with a `TableCell` using `rowSpan={tier.pokemon.length}`.
- Use shadcn/ui `Badge` or a small local chip component for effect chips.
- Use shadcn/ui `Collapsible` or controlled expandable rows for setup details.

Preferred: native table semantics through shadcn `Table`. Preserve grouped speed readability with `rowSpan`; use a separate stacked mobile layout only if the table becomes cramped.

## Expanded Details

Each Pokemon setup row may expand inline or open a side panel.

Expanded detail content:

```text
Raw speed: 154
Level: 50
IVs: 31
EVs: 252
Nature: +Spe
Formula: 154 x 2 Sand Rush = 308
Condition: sand
```

For stacked effects:

```text
Raw speed: 145
145 x 2 Chlorophyll = 290
290 x 1.5 Choice Scarf = 435
Condition: sun
```

If the frontend needs exact ordered calculation steps, add `steps[]` to emitted UI data later. Current grouped JSON does not include steps.

## Filters

Keep default table clean. Use filters to control clutter.

Initial filters:

- Search Pokemon name.
- Baseline only.
- Boosted only.
- Ability effects.
- Item effects.
- Weather condition: any, sun, rain, sand, snow.
- Spread: any, +Spe, neutral, -Spe.
- EVs: any, 252, 0.

Filter semantics:

```ts
baseline = effects.length === 0
boosted = effects.length > 0
ability = effects.some(effect => effect.kind === "ability")
item = effects.some(effect => effect.kind === "item")
weather = effects.some(effect => effect.condition === selectedWeather)
spread = pokemon.spread.nature === selectedNature
evs = pokemon.spread.evs === selectedEvs
```

Filtering should keep speed groups visible only if at least one child setup matches.

## Search

Pokemon search should:

- Match case-insensitive substrings.
- Search `name`.
- Keep matching setup rows only.
- Preserve speed sorting.

Optional later:

- Search effect labels and conditions.
- Search Pokedex number.

## Sorting

Default order:

- Speed groups descending.
- Pokemon rows use data order from ingestor.

Avoid client-side resorting within groups unless user requests it. Ingestor already emits deterministic order.

## Responsive Behavior

Desktop:

- Four-column table/list.
- Speed column compact.
- Effects wrap into multiple chip lines.
- Header can be sticky.

Mobile:

- Use grouped cards or compact row groups.
- Speed appears as group header.
- Pokemon, Spread, Effects stack inside each setup row.
- Avoid horizontal scrolling unless table mode explicitly chosen.

Mobile example:

```text
308
Excadrill
+Spe 252
[2x] [Sand Rush] [sand]
```

## Visual Direction

Aesthetic: precise, utilitarian, game-data terminal. Clean like a serious reference sheet, not decorative dashboard.

Color rule:

- Use only colors already present as CSS theme tokens in `apps/webapp/src/routes/layout.css`.
- Prefer token classes such as `bg-background`, `text-foreground`, `bg-card`, `text-muted-foreground`, `border-border`, `bg-primary`, `text-primary-foreground`, `bg-secondary`, `bg-muted`, `bg-accent`, `text-destructive`, and `ring-ring`.
- Do not add hard-coded hex, RGB, HSL, OKLCH, Tailwind arbitrary color values, or new CSS color variables.
- If implementation truly needs a new color, stop and ask for permission before adding it to `layout.css`.

Do:

- Use high contrast text.
- Use tabular numbers for speeds.
- Use restrained chip treatment by effect kind with existing theme tokens only.
- Use subtle group separators.
- Use row hover/focus states.
- Keep dense but breathable spacing.

Do not:

- Use large marketing hero.
- Use cards inside cards.
- Add new colors outside the `layout.css` theme.
- Use decorative orbs/blobs.
- Make effect chips so colorful they overpower speed/name.
- Hide key conditions behind hover-only UI.

Suggested chip token mapping:

```text
multiplier: bg-primary text-primary-foreground
ability: bg-secondary text-secondary-foreground
item: bg-accent text-accent-foreground
weather/condition: bg-muted text-muted-foreground
stage: text-destructive border-border
baseline: bg-muted text-muted-foreground
```

Use color as secondary signal. Text labels must carry meaning.

## Accessibility

- Table/list must be keyboard navigable.
- Expand controls need visible focus.
- Chips need readable text, not icon-only labels.
- Do not rely on color alone for effect kind.
- Use `aria-expanded` for expandable setup rows.
- Keep native table semantics through shadcn `Table`.
- If adding a separate mobile stacked layout, include accessible labels for Speed, Pokemon, Spread, Effects.

## Performance

Dataset can grow. Design for large lists.

Requirements:

- Avoid expensive per-render derived filtering if data grows; memoize filtered tiers.
- Keep chip rendering pure and simple.
- Consider virtualization only after measuring.
- Avoid client-side recomputation of speed math.
- Use generated `speed`, `spread`, and `effects` directly.

## Empty States

When filters hide all rows:

```text
No speed tiers match these filters.
```

When one speed group has no matching Pokemon after filtering, omit the group.

## Implementation Plan

2. Define frontend types matching ingestor contract.

4. Build filter state and `filterSpeedTiers`.
5. Build `SpeedTierTable`.
6. Build `SpeedGroup`.
7. Build `PokemonSetupRow`.
8. Build `EffectChips`.
9. Add expandable details.
10. Add responsive layout.

## Acceptance Criteria

- Default table shows columns: `Speed`, `Pokemon`, `Spread`, `Effects`.
- Same-speed Pokemon are grouped under one speed value.
- Baseline rows are visually distinct from boosted rows.
- Weather abilities show both ability and condition, e.g. `Sand Rush` + `sand`.
- Choice Scarf shows as item effect with `1.5x`.
- Stacked effects show all sources, not one combined vague modifier.
- Filters preserve grouped speed layout.
- Mobile layout remains readable without text overflow.
- No frontend code infers ability/item mechanics from raw slugs.

## Non-Goals

- Editing speed tier rows.
- Recalculating stats in the frontend.
- Full Pokemon set builder.
- Damage calculator.
- Team builder.
- Drag-and-drop spreadsheet behavior.
